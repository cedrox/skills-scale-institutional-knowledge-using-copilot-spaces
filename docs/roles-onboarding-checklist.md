# Roles Onboarding Checklist

Use this checklist when a new team member joins or when a PM needs to confirm that responsibilities, contacts, and access are properly set up for all roles on a project.

See [octoacme-roles-and-personas.md](octoacme-roles-and-personas.md) for full role definitions.

---

## All Roles (complete for every new team member)

- [ ] Role confirmed and agreed with PM and team lead
- [ ] Added to team communication channels (Slack/Teams/email groups)
- [ ] Added to project board (GitHub Projects or equivalent) with appropriate permissions
- [ ] Repository access granted (read / write / admin as appropriate)
- [ ] Key contacts identified: PM, PdM, Technical Lead, direct collaborators
- [ ] Welcome meeting or intro call scheduled with PM
- [ ] Project overview docs reviewed: [overview](octoacme-project-management-overview.md), [roles](octoacme-roles-and-personas.md)
- [ ] First sprint or milestone identified and backlog items assigned

---

## Project Manager (PM)

- [ ] Project Charter / One-pager reviewed and understood
- [ ] Risk register access confirmed
- [ ] Stakeholder list and communication plan reviewed
- [ ] Recurring cadences (standups, syncs, stakeholder updates) scheduled
- [ ] Escalation contacts (sponsor, Product Lead) confirmed

## Product Manager (PdM)

- [ ] Product roadmap and backlog access confirmed
- [ ] Success metrics and acceptance criteria reviewed
- [ ] Stakeholder alignment sessions introduced
- [ ] User research and analytics tool access granted

## Scrum Master

- [ ] Agile tooling access confirmed (project board, velocity tracking)
- [ ] Sprint ceremonies scheduled: daily standup, planning, review, retrospective
- [ ] Impediment escalation path agreed with PM (see [execution-and-tracking](octoacme-execution-and-tracking.md))
- [ ] Team norms and Definition of Done reviewed
- [ ] Access to team metrics and burndown charts confirmed

## Developers

- [ ] Repository access and branching conventions reviewed
- [ ] CI/CD pipeline access confirmed
- [ ] Local development environment set up and verified
- [ ] Code review and PR workflow understood
- [ ] Definition of Done reviewed

## QA / Testing

- [ ] Test environment access confirmed
- [ ] Automated test suite access and run instructions reviewed
- [ ] Smoke test plan reviewed (linked in [release-checklist.md](release-checklist.md))
- [ ] Bug tracking tool access confirmed
- [ ] Acceptance criteria process discussed with PdM and Developers

## Technical Lead

- [ ] Architecture and design decision records (ADRs) reviewed
- [ ] Codebase overview walkthrough scheduled with Developers
- [ ] Access to all relevant repositories and infrastructure dashboards confirmed
- [ ] Coding standards and PR review expectations communicated to the team
- [ ] Alignment meeting scheduled with PM and PdM on feasibility and estimates

## UX Designer

- [ ] Design tool access confirmed (Figma, Sketch, or equivalent)
- [ ] Design system and component library access confirmed
- [ ] User research data and past usability findings reviewed
- [ ] Collaboration workflow with Developers agreed (handoff process, annotation standards)
- [ ] Intro meeting with PdM to align on upcoming design priorities scheduled

## Release Engineer

- [ ] CI/CD pipeline access and credentials confirmed
- [ ] Release process and environment promotion flow reviewed
- [ ] [Release checklist](release-checklist.md) reviewed and customized for the project
- [ ] Rollback procedures and runbooks reviewed and accessible
- [ ] Deployment window and change management process understood
- [ ] Communication path to Customer Support Lead and PM confirmed

## Customer Support Lead

- [ ] Support tooling access confirmed (ticketing system, knowledge base)
- [ ] Product release timeline access confirmed
- [ ] Process for receiving release notes from Release Engineer confirmed
- [ ] Escalation path to PM and on-call engineering agreed
- [ ] Support readiness checklist integrated into [release-checklist.md](release-checklist.md)

## DevOps Engineer

- [ ] Cloud infrastructure access and IAM permissions confirmed
- [ ] Monitoring and alerting dashboards access confirmed
- [ ] Incident response runbooks reviewed and accessible
- [ ] Secrets management and access control policies reviewed
- [ ] Collaboration workflow with Release Engineer and Technical Lead agreed
